# Knowledge Update Policy

Purpose

Every conversation should improve project knowledge.

Conversation is temporary.

Knowledge is permanent.

---

Whenever new information appears, classify it.

Business Vision

-> context.md

New terminology

-> context.md (Project Vocabulary section)

Architecture

-> context.md (Architecture section)

Business Rules

-> context.md (Business Rules section)

Requirements

-> context.md (Current Priorities, Known Constraints, Current Decisions sections)

Major irreversible decision

-> ADR

Tasks

-> kanban

Agent behavior

-> questioning_policy.md / interview_framework.md

---

Never duplicate information.

Store information only once.

Reference instead of copying.

---

Update Rules

Only update documents when

A decision has been made.

Definitions become stable.

Requirements are confirmed.

Architecture changes.

Business rules change.

---

Do NOT update documents for

Ideas

Brainstorms

Speculation

Temporary discussion

Unconfirmed assumptions

---

Every update should answer

What changed?

Why?

Who decided?

Impact?

Files affected?

---

If knowledge conflicts with existing documents

Stop.

Report conflict.

Request clarification.

Never overwrite silently.

---

Knowledge Quality

Knowledge should be

Accurate

Minimal

Searchable

Non-duplicated

Stable

Easy to maintain

---

Every agent is responsible for maintaining a living knowledge base.
